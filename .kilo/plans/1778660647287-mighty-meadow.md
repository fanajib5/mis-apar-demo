# Rencana Arsitektur Produk: Multi-Mode Deployment + Multi-Tenancy

## Ringkasan

Produk APAR Management System akan diubah dari aplikasi demo tunggal menjadi **produk dengan 3 mode deployment** dan **2 model penjualan** (SaaS multi-tenant + White-label enterprise).

---

## 0. Perbandingan: Laravel Teams vs Multi-Tenant

| Aspek | Laravel Teams (Custom - Existing) | Tenant (Proposed) |
|-------|-----------------------------------|-------------------|
| **Tujuan** | User membership — siapa saja anggota suatu organisasi | Data isolation — data milik klien mana |
| **Cakupan** | Hanya tabel pivot `team_members` | SEMUA tabel data (customers, products, invoices, dll) |
| **Isolasi** | Tidak ada (shared DB, semua data visible) | Row-level (`tenant_id` WHERE clause) |
| **Relasi** | User ↔ Team (many-to-many) | Data ↔ Tenant (many-to-one) |
| **Route key** | Subdomain slug | Subdomain atau header |
| **Kompleksitas** | Ringan (custom concern, ~200 LOC) | Sedang (global scope + middleware) |
| **Paket** | Kustom (bukan third-party) | Kustom (bukan third-party) |

**Keduanya bisa hidup berdampingan:**
- **Team** = organisasi/perusahaan tempat user bekerja
- **Tenant** = batas data (client) — 1 team bisa punya 1 tenant, atau 1 tenant bisa = 1 instance

**Keputusan:** Pakai **Tenant kustom** untuk data isolation (tenant_id global scope).  
Laravel Teams tetap dipakai untuk user membership — tidak diubah.  
Spatie Permission Teams tetap **disabled** — permission bersifat global, bukan per-team.

## 1. Arsitektur Multi-Mode Deployment

```
┌──────────────────────────────────────────────────────────────────┐
│                        APAR Management System                      │
├──────────────────────────────────────────────────────────────────┤
│  Core App Layer (sama untuk semua mode)                          │
│  ┌─────────┬──────────┬──────────┬──────────┬────────────────┐   │
│  │ Auth    │ Master   │ Sales    │ APAR     │ Reports        │   │
│  │ (Team)  │ Data     │ (SO/Inv) │ (Servis) │ (Dashboard)    │   │
│  └─────────┴──────────┴──────────┴──────────┴────────────────┘   │
├──────────────────────────────────────────────────────────────────┤
│  Deployment Adapter Layer (berbeda per mode)                     │
│  ┌──────────┬──────────────┬──────────────┬──────────────────┐   │
│  │ Database │ Storage      │ Queue        │ Cache            │   │
│  │ Adapter  │ Adapter      │ Adapter      │ Adapter          │   │
│  └──────────┴──────────────┴──────────────┴──────────────────┘   │
├──────────────────────────────────────────────────────────────────┤
│  Multi-Tenancy Layer (SaaS/White-label)                          │
│  ┌──────────┬──────────┬──────────┬──────────────────────────┐   │
│  │ Tenant   │ License  │ Branding │ Feature Flags            │   │
│  │ Identity │ System   │ Override │ (per tenant module)      │   │
│  └──────────┴──────────┴──────────┴──────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

### Mode A: Full Online (SaaS)
| Layer | Implementation |
|-------|---------------|
| Database | PostgreSQL/MySQL (cloud) — Laravel Boot |
| Cache | Redis |
| Queue | Redis + Horizon |
| Storage | S3/Cloud (Laravel Flysystem) |
| Tenant | Single-tenant (sederhana) atau multi-tenant via subdomain |

### Mode B: Hybrid (Online + Offline with Sync)
| Layer | Implementation |
|-------|---------------|
| Database | PostgreSQL (cloud) + SQLite (local) |
| Cache | Redis (online) / File (offline) |
| Queue | Redis (online) / SQLite (offline buffer) |
| Storage | S3 (cloud) + Local (cache) — sync engine bidirectional |
| Sync Engine | Laravel Job + Conflict Resolution (last-write-wins) |

### Mode C: Full Offline (Local First)
| Layer | Implementation |
|-------|---------------|
| Database | SQLite only |
| Cache | File-based |
| Queue | Sync (no queue, process inline) |
| Storage | Local filesystem |
| Sync | Opsional — manual backup/restore, tidak untuk operasional |
| Delivery | **PHP Native wrapper** — PHP binary + SQLite + Nginx embedded, dibundle sebagai installer (Inno Setup / NSIS untuk Windows)

## 2. Multi-Tenancy Infrastructure

### Pilihan: Single DB + Tenant ID (Recommended untuk MVP)

Lebih sederhana, skalabilitas cukup untuk 1000+ tenant, dan kompatibel dengan semua mode deployment.

#### Files Baru:
```
app/Models/Tenant.php              — Model tenant (nama, domain, logo, config)
app/Models/Scopes/TenantScope.php  — Global scope untuk filter tenant_id
app/Http/Middleware/IdentifyTenant.php  — Middleware: deteksi tenant dari subdomain/header
app/Exceptions/TenantNotFoundException.php
```

#### Migration Baru:
```
database/migrations/xxxx_xx_xx_create_tenants_table.php
  - id, name, domain, database (nullable untuk multi-DB), settings (json)
  - logo_url, primary_color, custom_css (untuk white-label branding)

database/migrations/xxxx_xx_xx_add_tenant_id_to_existing_tables.php
  - ALTER TABLE ... ADD COLUMN tenant_id BIGINT UNSIGNED NOT NULL DEFAULT 1
```

#### Semua Model — tambahkan trait:
```php
use App\Models\Scopes\TenantScope;

protected static function booted(): void
{
    static::addGlobalScope(new TenantScope);
    static::creating(fn ($model) => $model->tenant_id ??= Tenant::current()?->id);
}
```

#### Middleware:
```php
// IdentifyTenant.php
// Priority:
//   1. Subdomain: tenant.app.example.com
//   2. Auth header: X-Tenant-ID
//   3. Session: tenant_id from login
//   4. Fallback: TENANT_ID from .env (single-tenant mode backward compat)
```

### Tenant Identification Strategy

| Method | Cocok untuk | Contoh |
|--------|-------------|--------|
| Subdomain | SaaS multi-tenant | `client-a.aparmanagement.com` |
| Header | White-label / API | `X-Tenant-ID: client_a` |
| ENV fallback | Single-tenant demo | `TENANT_ID=1` di .env |

## 4. Branding Overrides (White-Label)

```php
// AppServiceProvider::boot()
```

## 5. Sync Engine (Hybrid Mode B)

```
┌──────────────┐         ┌──────────────────┐
│   Local      │  sync   │   Cloud Server   │
│   (SQLite)   │◄───────►│   (PostgreSQL)   │
│   Offline    │  push/  │   Online         │
│   Queue      │  pull   │   Queue          │
└──────────────┘         └──────────────────┘

Conflict Resolution:
  1. **Merge logic** (bukan LWW sederhana) — gabungkan perubahan dari kedua sisi
  2. Jika ada konflik data yang sama diedit di dua tempat: **server menang** (konfigurasi)
  3. Log konflik di tabel sync_log untuk audit
  4. Admin dapat review dan resolve konflik manual via UI (opsional)
```

### Tabel Sync:
```
sync_entries: id, tenant_id, syncable_type, syncable_id, action (create/update/delete), payload (json), version, status (pending/synced/conflict), created_at, synced_at
```

### Sync Flow:
```
[INSERT di local SQLite]
  → Simpan ke sync_entries (status: pending)
  → Queue job: SyncToCloudJob
  → Job coba kirim ke endpoint POST /api/sync
  → Server terima, proses (create/update/delete)
  → Server balas status + server version
  → Jika sukses: ubah status jadi synced
  → Jika gagal: retry dengan backoff (Coolify queue)
```

### Files Baru:
```
app/Models/SyncEntry.php
app/Jobs/SyncToCloudJob.php
app/Jobs/SyncFromCloudJob.php
app/Services/SyncService.php
app/Http/Controllers/Api/SyncController.php
app/Http/Middleware/EnsureSyncAllowed.php
database/migrations/xxxx_xx_xx_create_sync_entries_table.php
routes/api.php — endpoint sync
```

### Optimasi Frontend (PWA):
```js
// Service Worker (resources/js/sw.js)
// Cache API responses → replay when online
// IndexedDB for local storage
// Register di app.tsx: navigator.serviceWorker.register('/sw.js')
```

## 10. Coolify Template per Mode

### Mode A (SaaS):
- Service type: Single container
- Build: nixpacks (existing)
- Service: nginx + php-fpm
- Queue: laravel-worker (additional service)
- Database: PostgreSQL (additional service)
- Cache: Redis (additional service)
- Storage: S3/MinIO (optional)

### Mode B (Hybrid):
- Service type: Single container (all-in-one) + Persistent volume
- Build: nixpacks + supervisor
- Services: nginx + php-fpm + queue-worker + sync-worker
- Database: PostgreSQL (cloud) + SQLite (local volume via persistent storage)
- Cache: Redis (additional service)
- Persistent volume: `/data` (untuk SQLite + offline buffer)

### Mode C (Offline):
- Tidak pakai Coolify (desktop app — PHP Native wrapper)
- Bundle: PHP binary + SQLite + Nginx embedded
- Delivery: Installer (Inno Setup untuk Windows)
- No cloud dependency — fully self-contained

## 11. Resiko & Mitigasi

| Resiko | Dampak | Mitigasi |
|--------|--------|----------|
| Spatie Permission + Multi-tenant conflict | Medium | Test scoping — Spatie teams disabled, pakai tenant_id manual di application layer |
| Sync conflict data loss (merge logic) | High | Log semua konflik, **server-win policy**, manual resolution UI untuk admin |
| Performance dengan 1000+ tenant single-DB | Medium | Index tenant_id + query optimization + pagination + caching |
| Test suite rusak karena tenant scope | Medium | Helper `Tenant::fake(1)` di semua test atau `withoutTenantScope()` scope method |
| White-label CSS injection security (XSS) | High | Sanitize CSS, gunakan CSS custom properties terbatas (hanya warna/font/logo URL) |
| PHP Native wrapper bundling | Medium | Riset tools: php-spin, static-php-cli, atau Laragon portable approach |
